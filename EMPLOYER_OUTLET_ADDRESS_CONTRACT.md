# Employer / Outlet Address Contract (Admin Web + Backend)

## Purpose
Ensure `hqAddress` (employer address) and `outlets[].address` (outlet address) stay independent and stable across:
- Admin Panel: `AddEmployer` / `EditEmployer`
- Backend: `/api/admin/employers` create/update + any persistence logic
- Native app rendering

This contract prevents address mix-ups where selecting an address in one field clears or overwrites the other.

---

## 1) Canonical fields

### Employer
- `hqAddress` (string, required)
  - The HQ / employer-level address
  - Must be stored exactly from `formData.hqAddress`

### Outlet
- `outlets[]` (array, optional)
  - Each outlet includes:
    - `name` (string)
    - `managerName` (string)
    - `contactNumber` (string)
    - `address` (string, required for any outlet you create/update)

---

## 2) Create Employer: `/admin/employers` (POST)

### Request body expectations
- `hqAddress` is read from request body and stored as employer’s address.
- `outlets[].address` is stored only on the outlet object it belongs to.

### Hard rule: no cross-writing
- Backend MUST NOT set `hqAddress` from any outlet field (`outlets[].address`, `outletAddress`, `location`, etc.)
- Backend MUST NOT set any outlet `address` from `hqAddress`
- Backend MUST NOT “default” missing outlet addresses to `hqAddress`

---

## 3) Update Employer: `/admin/employers/:employerId` (PUT)

### Independence rule
- Update `hqAddress` ONLY if the request explicitly includes a new `hqAddress`.
- Update `outlets[i].address` ONLY if that outlet object is included in the request payload for the update.

### Partial update safety
- If a particular outlet item is omitted from the update request (or filtered out by frontend because `address` is empty),
  - backend MUST keep the existing outlet address unchanged.

### Hard rule: no cross-writing
- Same as create:
  - Never overwrite `hqAddress` with an outlet address.
  - Never overwrite outlet `address` with `hqAddress`.

---

## 4) Employer ID behavior

- `employerId` is generated by the backend (immutable once created).
- Frontend may display:
  - “Auto-generated” prior to create
  - read-only `employerId` after fetch/edit

### Hard rule: id generation
- Backend MUST generate and persist `employerId` on create.
- Backend MUST NOT require `employerId` from the client for create/update.

---

## 5) Data normalization (optional but safe)

If normalization is required (e.g., trimming whitespace):
- Backend may `trim()` `hqAddress` and outlet `address`
- Backend MUST NOT remove the entire address or replace it with another field
- Backend MUST NOT convert outlet addresses into employer addresses

---

## 6) Example mapping (request -> persistence)

Request:
```json
{
  "hqAddress": "1 North Bridge Rd, Singapore",
  "outlets": [
    { "name": "Outlet A", "managerName": "Amy", "contactNumber": "9000...", "address": "10 Orchard Rd, Singapore" }
  ]
}
```

Persistence:
- Employer.hqAddress = `"1 North Bridge Rd, Singapore"`
- Outlet[0].address = `"10 Orchard Rd, Singapore"`

No other assignment is allowed.
